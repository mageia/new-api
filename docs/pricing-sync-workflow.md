# 定价同步工作流

> 目的：当上游 `pricing.json` 更新后，能快速对比本站当前定价配置，按 diff 逐项校准。

## 目标

- 保留一份可读的本站定价配置基线。
- 上游更新后，只比较 **enable 的模型**。
- 用 diff 明确哪些模型需要新增、调整、删除。
- 先更新本地文档，再同步到线上配置。

## 参考文件

- `docs/pricing.json`：上游原始参考。
- `docs/pricing.latest.json`：最近一次抓取的上游快照。
- `docs/pricing-calibration-report.md`：历史校准结论。
- `docs/pricing-calibration-plan.md`：历史处理思路。
- `docs/pricing-sync-workflow.md`：本工作流说明。

## 适用范围

只处理以下模型：

- 上游 `pricing.json` 中 `is_enabled = true` 的模型。
- 站内当前仍在使用、且需要保持计费一致的模型。

不处理的情况：

- 上游已禁用的模型。
- 本站未启用、也不计划启用的模型。

## 建议的更新顺序

1. 重新拉取最新上游 `pricing.json`，保存为 `docs/pricing.latest.json`。
2. 从 `docs/pricing.latest.json` 里筛出 `is_enabled = true` 的模型。
3. 与本站当前有效定价配置做 diff。
4. 只调整差异模型，不补无关模型。
5. 若是视频/图片阶梯计费，优先保留表达式，不要退化成固定价。
6. 更新后再做一次接口校验和前端展示校验。

## 上游价格获取方式

上游价格来自 Creative Studio 管理概览接口：

```bash
curl https://creative-studio.waninter.com/v1/admin/overview \
  --oauth2-bearer "$CREATIVE_STUDIO_ADMIN_TOKEN" \
  -o docs/pricing.latest.json
```

注意：

- 不要把真实 `sk_live...` Token 写入仓库、文档、脚本或命令历史。
- 建议在本地 Shell 中临时设置环境变量：

```bash
export CREATIVE_STUDIO_ADMIN_TOKEN='***'
```

- 每次同步前先覆盖更新 `docs/pricing.latest.json`，再与本站当前配置做 diff。

## Diff 规则

### 1. 新增模型

如果上游启用了新模型，本站也计划支持：

- 补模型定价配置。
- 同步模型可见性与分组限制。
- 检查前端是否展示正确的档位信息。

### 2. 价格变更

如果仅价格数值变化：

- 先更新本地文档快照。
- 再更新线上配置。
- 保持模型名、分辨率、时长、倍率等字段不变。

### 3. 阶梯计费变更

如果上游从固定价变成阶梯价，或阶梯档位变化：

- 不要强行压扁成单一价格。
- 用表达式或结构化计费配置保留原始规则。
- 需要时单独记录“请求参数 -> 档位 -> 价格”的映射。

### 4. 删除模型

如果上游模型已禁用，且本站也不再提供：

- 从本站定价配置中删除。
- 如果已有旧渠道或旧模型入口，要同步禁用或下线。
- 避免模型广场继续显示已经不支持的模型。

## 本站推荐的配置表达

### 图片模型

- 优先保留 `size` / `quality` 对应的阶梯表达。
- 如果某模型只有固定价，再用固定价。
- 不要因为实现方便，丢掉上游档位差异。

### 视频模型

- 优先保留 `duration` / `resolution` / `sample_count` 相关表达。
- 只在上游确实是单档固定价时，才使用固定价。
- 已经确认的私有分组，继续保留分组倍率差异。

## 更新检查清单

- 上游快照是否已更新到 `docs/pricing.latest.json`。
- 仅对 `is_enabled = true` 的模型做对比。
- 站内配置是否存在未对齐的旧模型。
- 旧渠道是否仍在暴露已废弃模型。
- 前端“模型广场”和“创建令牌”是否展示一致。
- 价格展示和扣费逻辑是否一致。

## 推荐产出物

每次更新建议保留两份文档：

1. `docs/pricing.latest.json`：上游最新快照。
2. `docs/pricing-sync-workflow.md`：流程说明。

如果需要人工复盘，再补一份只读报告，例如：

- 哪些模型新增。
- 哪些模型改价。
- 哪些模型下线。
- 哪些模型只改了分组可见性。

## 结论

需要这个文档。它的作用不是存“最终价格”，而是作为每次上游变更后的**对照流程**：

- 上游更新时有据可查。
- 校准时能按 diff 精确修改。
- 旧模型、废弃渠道、分组可见性问题更容易统一排查。
